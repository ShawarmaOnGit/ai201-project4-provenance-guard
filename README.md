# Provenance Guard

This is a backend API for classifying whether submitted text was written by a human or generated by AI. Built for creative sharing platforms that want to surface transparent attribution labels, handle creator appeals, and maintain a structured audit trail.

---

## Architecture Overview

The path a submission takes from input to the label a user sees:

```
POST /submit
    │
    ├── Input validation (text ≥ 20 chars, creator_id present)
    │
    ├── Signal 1: LLM Classification (Groq llama-3.3-70b-versatile)
    │     → ai_probability float [0.0, 1.0]
    │
    ├── Signal 2: Stylometric Heuristics (pure Python)
    │     → stylo_score float [0.0, 1.0]
    │
    ├── Confidence Scorer
    │     ai_prob = 0.60 × llm + 0.40 × stylo
    │     → attribution (likely_ai / uncertain / likely_human)
    │     → confidence score
    │
    ├── Label Generator
    │     → plain-language label string
    │
    ├── Audit Log (audit_log.jsonl)
    │     → appends structured JSON entry
    │
    └── JSON Response → caller


POST /appeal
    │
    ├── Lookup content_id in content store
    ├── Update status → "under_review"
    ├── Audit Log → append appeal_submitted event
    └── JSON Response → caller
```

## Detection Signals

### Signal 1: LLM-Based Classification

**Tool:** Groq `llama-3.3-70b-versatile`

**What it measures:** Holistic semantic and stylistic patterns. The model evaluates whether the text reads as genuinely human: irregular emotional texture, natural imperfections (contractions, casual phrasing, run-ons), inconsistent formality, and an idiosyncratic personal voice. AI writing tends to be fluent, consistently formal, and free of the small uncertainties that appear in spontaneous human composition.

**What it misses:** Formally trained humans (academics, lawyers, technical writers) may score high because their natural register lacks casual markers. AI prompted to write colloquially may score low.

---

### Signal 2: Stylometric Heuristics

**Tool:** Pure Python (no external libraries)

**What it measures:** Four measurable structural properties:

1. **Sentence length uniformity** — Coefficient of variation of sentence word-counts. AI text tends to produce similarly-sized sentences; human writing has higher variance.
2. **Filler/hedge word absence** — Presence of words like "honestly," "kinda," "I guess," "tbh," "dunno." Human writing uses these regularly; AI almost never does.
3. **AI transition and buzzword density** — Frequency of phrases like "Furthermore," "paradigm shift," "stakeholder," "nuanced," "it is important to note." High density signals AI authorship.
4. **Type-token ratio (TTR)** — Unique words / total words. AI text clusters in a mid-range (0.55–0.78); human writing is either more repetitive (casual speech) or more varied (rich creative prose).

**What it misses:** Academic and technical human writing has low filler density and uses formal transitions legitimately. Very short texts (< 3 sentences) reduce the reliability of sentence-length variance.

## Confidence Scoring

### How Signals Are Combined

```
ai_probability = 0.60 × llm_score + 0.40 × stylo_score
```

The LLM signal receives more weight (60%) because it captures semantic and holistic stylistic patterns that statistical heuristics cannot — it is the stronger signal overall. The stylometric signal provides an independent structural check that cannot be influenced by the LLM's prompt or temperature.

### Threshold Map

| `ai_probability` | Attribution | Confidence |
|---|---|---|
| ≥ 0.70 | `likely_ai` | `= ai_probability` |
| ≤ 0.35 | `likely_human` | `= 1.0 − ai_probability` |
| 0.36 – 0.69 | `uncertain` | `= \|ai_prob − 0.525\| / 0.175` |

The uncertain band is intentionally wide. A false positive — labeling a human's work as AI-generated — damages a creator's reputation. The system only commits to `likely_ai` when `ai_probability ≥ 0.70`, requiring strong evidence from both signals.

### Validation: Two Example Submissions with Different Scores

**Example 1 — High-confidence AI-generated**

Input text:
```
Artificial intelligence represents a transformative paradigm shift in modern society.
It is important to note that while the benefits of AI are numerous, it is equally
essential to consider the ethical implications. Furthermore, stakeholders across
various sectors must collaborate to ensure responsible deployment.
```

| Signal | Score |
|--------|-------|
| LLM score | 0.90 |
| Stylometric score | 0.5854 |
| **Combined ai_probability** | **0.7742** |
| **Attribution** | **likely_ai** |
| **Confidence** | **0.7742** |

The LLM correctly identifies formal structure, buzzwords ("paradigm shift," "stakeholders"), and the absence of personal voice. The stylometric signal finds 4 AI phrase hits in 3 sentences, no filler words, and a uniformity score of 0.38.

---

**Example 2 — High-confidence human-written**

Input text:
```
ok so i finally tried that new ramen place downtown and honestly?
underwhelming. the broth was fine but they put WAY too much sodium in it and
i was thirsty for like three hours after. my friend got the spicy version and
said it was better. probably wont go back unless someone drags me there
```

| Signal | Score |
|--------|-------|
| LLM score | 0.08 |
| Stylometric score | 0.0642 |
| **Combined ai_probability** | **0.0737** |
| **Attribution** | **likely_human** |
| **Confidence** | **0.9263** |

The LLM correctly identifies casual lowercase, incomplete punctuation, colloquial phrasing, and natural personal voice. The stylometric signal finds 3 filler-word hits ("honestly," "like"), 0 AI phrases, and high sentence-length variance (CV = 0.68).

The difference between the two examples — confidence 0.77 vs 0.93 in opposite directions — demonstrates that the scoring produces meaningful variation, not a constant output.

## Transparency Label

The label returned in the `/submit` response is the text that would be shown to a reader on the platform. All three variants use plain language — no jargon like "classifier output" or "logit score."

### Variant 1: High-Confidence AI-Generated

> "AI-Generated Content (77% confidence): Our system's analysis strongly suggests this content was produced by an AI writing tool. Multiple signals — including writing style, sentence structure, and vocabulary patterns — indicate AI authorship. If you are the creator and believe this is a mistake, you can submit an appeal."

### Variant 2: High-Confidence Human-Written

> "Human-Written Content (93% confidence): Our system's analysis suggests this content was written by a person. The writing shows natural variation, personal voice, and characteristics consistent with human authorship. If you believe this classification is incorrect, you can submit an appeal."

### Variant 3: Uncertain

> "Origin Unclear: Our system could not confidently determine whether this content was written by a human or generated by AI. The writing contains a mix of characteristics from both (54% AI indicators, 46% human indicators). This content has not been marked either way — the creator may submit an appeal to provide additional context for reviewers."

## Rate Limiting

**Limits applied to `POST /submit`:**
- **10 requests per minute**
- **100 requests per day**

**Reasoning:**

A realistic creative writing platform user submits a piece of work when they finish it — not on a script. A prolific writer might submit a new poem or essay a few times per day. The daily cap of 100 is generous for any single author while preventing automated bulk-scanning.

The per-minute cap of 10 protects against two adversarial patterns: (1) a creator flooding the system with variations of their content to probe the classifier's decision boundary, and (2) a malicious actor submitting thousands of pieces through the API to map the model's weaknesses. Ten submissions per minute allows natural usage (a developer testing their integration) while cutting off scripted attacks.

When the limit is exceeded, the server returns HTTP 429:

```json
{
  "error": "Rate limit exceeded",
  "message": "Too many submissions. Limit: 10 per minute, 100 per day. Please wait before submitting again.",
  "retry_after": "..."
}
```

## Known Limitations

**Non-native English speakers writing in a formal register** are the most likely source of false positives. A creator whose first language is not English often writes in a structured, careful style — avoiding contractions, using formal transitions legitimately, and composing grammatically complete sentences. The stylometric signal scores their filler-word absence and sentence uniformity as AI-like because those are genuine statistical features of AI text — but they are also genuine features of careful formal human writing. The LLM signal may also score them high because formal register looks "AI-generated" to a model trained primarily on informal web text. This limitation is tied directly to the signals chosen: both signals distinguish AI text from *casual* human text better than from *formal* human text. The wide uncertain band and appeal workflow are the primary mitigations; a human reviewer seeing the creator's stated reasoning can correct the classification.

## Spec Reflection

**One way the spec helped:** The thresholds table in the Uncertainty Representation section (planning.md) was written before any code. When implementing `compute_confidence()`, having the exact cutoffs (≥ 0.70 = AI, ≤ 0.35 = human) already decided made the function straightforward to write and verify — there was no ambiguity about where the boundary should sit. Running the four test inputs against those thresholds confirmed they produced the expected label distribution without needing to adjust them.

**One way implementation diverged:** The planning spec described Signal 2's TTR scoring as a simple threshold ("mid-range TTR → AI"). During implementation, this produced a cliff-edge at the threshold boundaries — a text with TTR = 0.54 would score dramatically differently from one at 0.56. The implementation replaced the step function with a Gaussian curve centered at 0.65, producing a smooth score that is highest for mid-range TTR and tails off for both very low and very high values. This better captures the intuition that mid-range TTR is an indicator, not a hard diagnostic, and prevents erratic scoring for borderline inputs.

## AI Usage

### Instance 1: Planning and implementing the Flask skeleton and LLM signal

I provided Claude with the Detection Signals section from planning.md and the architecture diagram (submission flow) and asked it to generate: (1) the Flask app skeleton with `POST /submit` and `GET /log` route stubs, and (2) a `llm_signal(text: str) -> float` function using Groq's chat completions API. Claude produced a solid structure and a well-crafted prompt for the Groq model. I verified the output by tracing through the prompt and function signature manually.

What I revised: The initial implementation used `json.loads(response_content)` directly. When I tested it, I discovered that `llama-3.3-70b-versatile` sometimes wraps its JSON in a markdown code block (` ```json ... ``` `). I added a `re.search(r"\{.*?\}", raw, re.DOTALL)` regex extraction before parsing, so the function handles both clean JSON and markdown-wrapped JSON responses. I also added the explicit `temperature=0.1` parameter to reduce LLM variability in the classification output — Claude's initial version omitted this.

### Instance 2: Planning and implementing the stylometric signal and confidence scoring

I provided Claude with the Detection Signals (Signal 2 sub-signals and weights) and Uncertainty Representation (threshold table) sections from planning.md and asked it to generate: (1) `stylometric_signal(text: str) -> float` implementing the four sub-signals, and (2) `compute_confidence(llm_score, stylo_score) -> dict` applying the threshold table.

What I revised: Claude's initial TTR implementation used a simple piecewise step function: `if 0.4 < ttr < 0.8: score = 0.7 else: score = 0.2`. Testing with the four sample inputs showed that this created a sharp discontinuity — the score jumped from 0.2 to 0.7 the moment TTR crossed 0.4, making borderline texts score erratically. I replaced it with a Gaussian curve `0.40 + 0.55 * exp(-((ttr - 0.65)² / (2 × 0.12²)))` centered at 0.65, which peaks there and falls off smoothly. I also adjusted the sub-signal weights: Claude's version had uniformity at 40% and transition density at 20%, but after testing I found transition density was more discriminating (AI phrase presence is a very strong signal) and rebalanced to 30%/30%.
